**The C++ in this project must be ultra-clean — no spaghetti.** Before writing or modifying any `.hpp`/`.cpp`, read the C++ style guide at the workspace root **in full**: `cpp-style-guide.md` (absolute path: `c:\Users\sivuc\projects\ChessAIv2\cpp-style-guide.md`). It is authoritative and not to be improvised around.

**The standard, condensed.** Small single-purpose functions (soft ceiling ~60 lines); shallow control flow via early-return guard clauses (nesting ≤ ~3); const-correct throughout (`const`, `const&`, `constexpr`, `const` methods); `constexpr` constants never `#define`; no magic numbers; `enum class` over plain enum; no raw owning pointers or `new`/`delete` (use values, `std::unique_ptr`, `std::vector`, stack); `static_cast` never C casts; `nullptr` never `NULL`; no `using namespace` in a header. **Zero heap allocation, I/O, or locks on the per-node search path** — this is a performance engine; measure speed with `bench` and strength with SPRT, never by feel.

**Match the existing conventions exactly** (see `src/types.hpp`, `src/eval.*`, `src/search.cpp` as the golden examples): `#pragma once`; all first-party code in `namespace engine { … } // namespace engine` with matching close-comments; internal-linkage helpers in an anonymous namespace in the `.cpp`; PascalCase types, camelCase functions and locals, trailing-underscore member variables (`nodes_`), `SCREAMING_SNAKE_CASE` `constexpr` constants; 4-space indent, K&R attached braces, 100-column lines, West const, pointers/refs bound to the type (`const Board&`). The only sanctioned `snake_case` functions are the idiomatic `types.hpp` helpers (`mate_in`, `mated_in`, `is_mate`) and the `run_uci` entry shim.

**Formatting is machine-defined.** Apply `.clang-format` (repo root) to every file you touch before finishing: `clang-format -i src/<file>.hpp src/<file>.cpp`. A curated `.clang-tidy` additionally flags bug-proneness/performance/Core-Guidelines issues (advisory; `clang-tidy -p build src/*.cpp`). These enforce layout and a slice of semantics only — naming, single-responsibility, ownership, and hot-path discipline are on you. A style checker, `scripts/check_cpp_style.py`, is also wired into the Claude Code hooks and **blocks** on the unambiguous mechanical violations (tab indentation, header missing `#pragma once`, `using namespace` in a header); check locally with `python scripts/check_cpp_style.py --check`.

**Use the vendored `include/chess.hpp` correctly** (it is third-party — do not modify or document it): `chess::Board`/`Move` directly, `movegen::legalmoves`, make/unmake, incremental `board.hash()` (never `board.zobrist()` in search), and avoid `board.isGameOver()` inside search.

If a rule seems wrong for a specific case, **ask** — do not assume, improvise, or create your own exception. Note also the always-on rule that **every code change requires a matching `.md` documentation change in the same turn** (see `documentation-style-guide.md` and `.agents/rules/documentation.md`); a C++ change is done only when it is both clean per `cpp-style-guide.md` and its companion is updated. The equivalent always-on rule for Claude Code lives in `CLAUDE.md`.
